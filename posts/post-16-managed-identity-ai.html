Managed Identity for AI: No API Keys, Ever

API keys are the most common security debt in AI workloads. They get committed to repositories, included in logs, shared across environments, and never rotated. Managed identity eliminates the key entirely. The identity is attached to the compute resource, not stored anywhere, and Azure handles credential rotation automatically. For every Azure AI service that supports managed identity - which is all of them - there is no justification for using a key in production.

Why API Keys Are Dangerous

An API key is a static credential with no expiry enforcement, no rotation guarantee, and no per-call identity. If it appears in a log, a repository, an environment variable file, or a container image layer, the blast radius is the entire deployment it was issued for. Keys are also shared across environments by default unless someone explicitly issues separate keys per environment - and many teams do not. Managed identity replaces all of this with a per-resource cryptographic identity that Azure issues and rotates automatically.

System-Assigned vs User-Assigned

TypeLifecycleWhen to Use
System-assignedTied to the resource - deleted when the resource is deletedSimple cases where one compute resource calls one AI service; lifecycle coupling is acceptable
User-assignedIndependent resource - survives compute deletion and can be sharedMultiple compute resources calling the same AI service; blue/green deployment patterns; lifecycle independence required

For most AI production workloads, user-assigned is the better default. When you redeploy a Container App or rotate a Function App instance, the identity persists and role assignments survive. You do not need to re-grant RBAC after every deployment.

DefaultAzureCredential and the Credential Chain

DefaultAzureCredential is the correct client-side pattern for application code. It tries credential sources in order: environment variables, workload identity federation, managed identity, Azure CLI, Visual Studio Code, and the Azure Developer CLI. The same code works locally using your az login session and in production using managed identity - no code branch, no environment variable flag, no conditional credential selection.

The only setup required in code is removing the key-based client constructor and replacing it with a DefaultAzureCredential-based one. For the Azure AI SDKs, that means passing a credential object rather than an API key string to the client constructor.

RBAC Roles for AI Workloads

  • Cognitive Services OpenAI User: call inference endpoints; the minimum role for applications calling Foundry model deployments
  • Azure AI Developer: create and manage Foundry resources within a project; needed for applications that create agent threads or manage evaluations
  • Search Index Data Reader: query Azure AI Search indexes; for application components that retrieve grounding documents
  • Search Index Data Contributor: read and write index data; for indexing pipelines that push documents into AI Search
  • Storage Blob Data Reader: read from blob storage; for document ingestion pipelines pulling source files

Assign roles at the smallest scope that covers the requirement. A Container App that only calls a single Foundry deployment should have Cognitive Services OpenAI User on that deployment, not on the hub or subscription.

Wiring Managed Identity in Practice

Container Apps: assign a user-assigned managed identity to the Container App environment or individual app, grant that identity the required RBAC roles on the target AI resources, and use DefaultAzureCredential in application code. No secrets, no environment variables containing credentials.

AKS: use Azure AD Workload Identity. Federate a Kubernetes service account to an Azure managed identity, annotate pods with the service account, and DefaultAzureCredential resolves the workload identity token automatically at runtime.

Local development: run az login before starting your application. DefaultAzureCredential picks up the CLI session. Never use production keys locally; if a developer needs to test against a production model deployment, grant their personal Entra identity the required role with an expiry date.

Key principle: If you have an API key in a Key Vault secret being injected into an environment variable, you have a managed identity migration to do.
