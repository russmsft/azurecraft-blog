Microsoft Foundry: The New AI Platform Model

Microsoft Foundry is the unified AI platform that replaces the old pattern of provisioning standalone Azure OpenAI resources. If you are building anything AI-related on Azure today, your starting point is a Foundry hub and project, not a raw OpenAI endpoint. This post walks through the resource model, what lives inside a project, and the deployment choices that matter for production workloads.

Hubs and Projects: Get the Ownership Model Right Early

Foundry introduces two resource tiers. Retrofitting the ownership model after deployment is painful, so sort this out before you write the first Bicep file.

A hub is the shared infrastructure layer: virtual network integration, managed identity, storage accounts, and Key Vault connections. One hub typically spans an entire platform team or business unit. Network security signs off once at the hub level, not per project team.

A project is a team workspace that lives inside a hub. Projects own deployments, agent configurations, evaluations, and connections to downstream services. A team of three building a customer support assistant gets their own project. They inherit the hub's network and identity without negotiating VNet peering themselves.

  • Hub: shared network, managed identity, and storage - owned by the platform or ops team
  • Project: model deployments, Foundry Agent Service, connections, evaluations - owned by the product team
  • One hub supports many projects; projects cannot span hubs
  • Role assignments at hub level cascade to projects; project-level roles are additive

What a Foundry Project Contains

A project is not just a namespace. It is the deployment unit for everything a team needs to ship an AI feature.

  • Model deployments: GPT-4.1, o4-mini, Phi-4, or any model available in your region and tier
  • Foundry Agent Service: the runtime for prompt-based and hosted agents
  • Connections: typed references to Azure AI Search indexes, storage accounts, Cosmos DB, and external APIs
  • Evaluations: built-in eval pipelines tied to your deployed models and agent runs
  • Prompt flows: for teams that need orchestration without writing SDK code

Everything is scoped to the project. Your team's model deployment does not bleed into another team's quota. Evaluations run against the same deployment they were built for. Connections are reusable across agents in the same project without passing secrets through environment variables.

Why Standalone Azure OpenAI Is an Anti-Pattern Now

Standalone Azure OpenAI resources predate the Foundry model. They still work. But they were designed for a world where you needed one model and one API key. That world is gone.

Build outside a Foundry project today and you lose:

  • Centralized connection management - each service manages its own secrets
  • Built-in evaluation hooks - you wire your own eval pipeline from scratch
  • Foundry Agent Service - it only operates within a project
  • Unified monitoring and tracing through the Foundry portal

The migration path from standalone resources is straightforward: create a hub, create a project, move deployments into the project. Existing quota follows the subscription, not the resource. The harder part is updating every service that holds a direct AOAI endpoint URL.

ProvisionedHigh-volume, predictable traffic, latency-sensitive pathsLowest and most consistentReserved capacity billed in PTUs

Provisioned throughput units make sense once you can justify the baseline commitment. Run the math: if your projected token volume at Global Standard rates exceeds the PTU reservation cost over 30 days, provision. Most teams start on Global Standard and move select deployments to Provisioned after they have real production traffic data to benchmark against.

Key principle: If your architecture diagram still shows a standalone "Azure OpenAI resource", update the design.
