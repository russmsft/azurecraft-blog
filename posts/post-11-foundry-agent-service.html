Foundry Agent Service: Prompt Agents vs Hosted Agents

Foundry Agent Service gives you two fundamentally different agent types, and picking the wrong one costs weeks of engineering time. Prompt-based agents need no custom code and cover the majority of real-world use cases. Hosted agents run your containerized logic with full control over the agent loop. Here is how to decide which one you actually need - and when not to reach for the more complex option.

The Core Distinction

A prompt-based agent is configured entirely in the Foundry portal. You write a system prompt, attach tools, point it at a knowledge source, and deploy. No container. No SDK code. The Foundry runtime handles conversation state, tool dispatch, and response assembly. The model decides which tools to call and in what order.

A hosted agent runs your code in a container. You use the Microsoft Agent Framework, LangGraph, or any framework you choose. Foundry manages the hosting, scaling, and observability surface. You own the agent loop: tool call sequencing, retry logic, branching, human-in-the-loop checkpoints.

The decision is simpler than most teams make it: if you are writing code to control when and how tools get called, you need a hosted agent. Everything else is a prompt-based agent.

When Prompt-Based Is the Right Call

Prompt-based agents handle the vast majority of production assistant use cases. They are not training wheels for beginners - they are the production-grade choice when nondeterministic orchestration is acceptable:

  • RAG-backed Q&A assistants over internal documentation with Azure AI Search
  • Customer support agents with Bing grounding for current product and pricing information
  • Structured extraction workflows where the model decides which tools to invoke based on context
  • Multi-turn conversational assistants with file upload and analysis capability
  • Agents that call Azure Functions or third-party REST APIs via OpenAPI specs

When Hosted Agents Are Worth the Overhead

Hosted agents add real complexity. Three situations justify that tradeoff:

  • Deterministic pipelines where tool call sequence must be explicit, auditable, and reproducible across runs
  • Custom orchestration frameworks you are already committed to - LangGraph, custom state machines
  • Agent loops that need embedded business logic: retry thresholds, conditional branching, or mandatory human approval before continuing

If your justification is "we want more control," that is not a hosted agent requirement. That is a system prompt and evaluation problem.

If your agents will handle customer data, proprietary documents, or anything under a compliance framework, start with Standard. The setup cost is an afternoon. Retrofitting it after you have live thread history is significantly more.

Connected Agents: Wiring Specialists Without an Orchestrator

Foundry connected agents let you wire multiple prompt-based agents together without custom routing code. One agent acts as the coordinator. Specialist agents register as tools it can delegate to. The coordinator routes based on conversation context - no intent classifier, no routing rules file, no custom message bus.

Tools available to any Foundry agent: code interpreter, file search, Bing grounding, Azure Functions, and OpenAPI specs for third-party REST APIs.

Key principle: Start with a prompt-based agent. Only reach for hosted agents when you need code-level control over the agent loop.
